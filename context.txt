Current CSV partitioning
- Current builder groups by:
  Mail group (FMS/AFS)
  VendorNum
  Currency
- It does NOT currently split by CompanyCode.
- Naming is driven by classify_mail_group(company_code) in src/processing/logic.py

Mail-group classification rule
- FMS:
  1000, 2000, E100
- AFS:
  0032, 0016, 0133, 0060, 1010, 5500, 0224
  and any company code starting with E except E100
- Current implementation in logic.py:
  classify_mail_group(company_code)

Business rules already implemented
- Invoice Number:
  uses "Invoice Number", fallback to ticket Id.
- Invoice Date:
  uses "Invoice Date", fallback to Created from HDA grid.
- VendorNum:
  E100 -> 900000
  1000/2000 -> 900010
  else -> 8000001
- Cost center normalization:
  removes hyphens/spaces.
  if numeric after cleaning -> pad to 10 digits.
  if alphanumeric -> keep cleaned value.
- Profit/Cost Center routing based on Account:
  If Account starts with 11, 12, 13, P1, P2, P3:
    center goes to Profit Center 10 DIGITS
    Cost Center 10 DIGITS left blank
  If Account starts with 14, 15, 16, P4, P5, P6:
    center stays in Cost Center 10 DIGITS
    Profit Center blank
  Else:
    current builder defaults to Cost Center

ZIP / address rule status
- Reverted experimental override that forced OH -> ON in Canadian context.
- Current policy:
  prioritize what the ticket/address actually says.
- Example:
  "Mason OH 45040" remains OH / 45040.
- Team confirmed that was correct for now.

Known weird address case
- Some tickets may be company/currency Canada but still contain a US-looking address/ZIP.
- Current decision:
  do not auto-correct those based on business context alone.
  preserve what the ticket says.

SAP knowledge gathered so far
- System split:
  FMS -> RP1
  AFS -> LUP
- TCode:
  ZFIN_AP_NONPO_LUCY4
  (confirmed same for AFS and FMS)
- Posting date:
  always current day
  can be typed like MMDDYYYY, e.g. 04092026
- Separator:
  must be ","
- Test checkbox:
  MUST be selected
  very important, otherwise posting could happen.
- Validation is run one CSV at a time.
- SAP does NOT care much about the CSV filename.
- For test execution, the form must have coherent values (system-related, company code field filled, currency filled).
- Company code field must be filled, but Ingrid indicated the test can still run even if CSV has multiple company codes; SAP is not using that field strictly as a partition gate in test mode.
- Result interpretation:
  look at the middle result table
  last column is message
  same source row can appear repeated if multiple errors exist

Interpretation from meeting:

FMS table

Interpretation:

Important SAP result behavior
- One logical CSV row may produce multiple result rows if SAP returns multiple messages.
- In the inspected AFS run, all rows had LINE_NO=0 and the same XBLNR=735105G, but multiple error messages.
- Therefore the result parser should group rows by a stable key, likely:
  XBLNR
  and maybe LINE_NO as secondary

Expected final Process 1 flow
1. Login HDA
2. Read all OneTime Check tickets
3. Run local validation
4. Reject obviously invalid local rows later (not implemented yet)
5. Generate CSV candidates
6. Run SAP test validation on each candidate CSV
7. Parse SAP result table
8. Remove bad rows / reject corresponding tickets
9. Rebuild/retest until SAP clean
10. Consolidate final delivery CSVs if needed
11. Send final emails
